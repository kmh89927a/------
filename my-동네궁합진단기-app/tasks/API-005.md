## 1. 🎯 Summary

- **기능명:** [API-005] SavedSearch 도메인 Request/Response DTO 정의 — saveSearch(), getSavedSearch() *(Rev 1.1: replaySearch 제거)*
- **목적 (Why):**
  - **비즈니스:** 반복 이사자(C-04)의 간이 저장·불러오기 기능의 API 통신 계약을 확립하여, CMD-SAVE-001(저장)과 QRY-SAVE-001(불러오기) 구현의 타입 기반을 마련한다.
  - **사용자 가치:** 저장·복원 요청/응답의 타입 안전성이 보장되어, best effort 저장 실패 시에도 예측 가능한 동작을 제공한다.
- **범위 (What):**
  - ✅ 만드는 것: saveSearch Request/Response DTO, getSavedSearch Response DTO, Zod 유효성 스키마, Prisma→DTO 변환 함수, 에러 코드
  - ❌ 만들지 않는 것: replaySearch DTO(제거됨), 비교 뷰 타입(REQ-FUNC-026 제거), 시나리오 비교 타입(REQ-FUNC-027 제거), SavedSearch CRUD 로직(CMD-SAVE 범위)
- **복잡도:** L
- **Wave:** 2 (API Contract 트랙)

- [ ] **3.5** `lib/mappers/saved-search-mapper.ts`에 Prisma → DTO 변환 함수
  ```typescript
  import type { SavedSearch as PrismaSavedSearch } from '@prisma/client';
  import type { SavedSearchDTO, SearchParams } from '@/lib/types/saved-search';
  import type { GetSavedSearchResponse } from '@/lib/types/saved-search-api';

  export function mapSavedSearchToDTO(record: PrismaSavedSearch): SavedSearchDTO {
    return {
      userId: record.userId,
      searchParams: record.searchParams as SearchParams,
      savedAt: record.savedAt,
    };
  }

  export function mapToGetResponse(
    record: PrismaSavedSearch | null,
    geocodingStatus?: { addressAValid: boolean; addressBValid: boolean }
  ): GetSavedSearchResponse {
    if (!record) {
      return { found: false, savedSearch: null };
    }
    return {
      found: true,
      savedSearch: mapSavedSearchToDTO(record),
      geocodingStatus: geocodingStatus ? {
        ...geocodingStatus,
        message: (!geocodingStatus.addressAValid || !geocodingStatus.addressBValid)
          ? '주소를 다시 입력해주세요' : undefined,
      } : undefined,
    };
  }
  ```

## 4. ✅ Acceptance Criteria (GWT 패턴, BDD)

**AC-1 (정상):** saveSearch Request Zod 검증 통과
- **Given** 유효한 `searchParams` (서울 강남구 주소, `maxCommuteTime: 60`, `mode: 'couple'`) 포함 요청
- **When** `saveSearchRequestSchema.parse(request)` 호출
- **Then** 검증 통과, 파싱된 객체의 `searchParams.mode`가 `'couple'`

**AC-2 (예외):** searchParams 누락 시 Zod 에러
- **Given** `searchParams` 필드가 없는 요청 객체
- **When** `saveSearchRequestSchema.parse(request)` 호출
- **Then** ZodError 발생, `issues[0].path`가 `['searchParams']`

**AC-3 (예외):** 저장된 기록 없음 시 응답 구조 검증
- **Given** DB에 해당 `userId`의 SavedSearch 레코드가 없는 상태
- **When** `mapToGetResponse(null)` 호출
- **Then** `found === false`, `savedSearch === null`, `geocodingStatus === undefined`

**AC-4 (경계):** geocoding 실패 시 안내 메시지 포함
- **Given** SavedSearch 레코드 존재하나 저장된 주소A의 geocoding이 실패
- **When** `mapToGetResponse(record, { addressAValid: false, addressBValid: true })` 호출
- **Then** `geocodingStatus.addressAValid === false`, `geocodingStatus.message === '주소를 다시 입력해주세요'`

**AC-5 (정상):** 에러 코드 전수 매핑 검증
- **Given** `SavedSearchErrorCode` enum의 모든 값
- **When** 각 값에 대해 `SAVED_SEARCH_ERROR_MAP[code]` 조회
- **Then** 모든 enum 값에 매핑이 존재, `message`가 한국어, `httpStatus`가 유효 범위

## 9. 🚧 Open Questions / Risks (보류 사항)

1. **userId 전달 방식:** `SaveSearchRequest`에 `userId`를 포함하지 않고 서버 사이드에서 Supabase Auth 세션으로 추출하는 설계. 클라이언트에서 userId를 직접 전달할 필요가 있는 케이스가 있는지 CMD-SAVE-001 작업 시 확정.
2. **SearchParams 스키마 진화:** DB-006의 Open Question과 동일 — `searchParamsSchema`를 `passthrough()` vs `strict()` 모드 결정 시 본 DTO의 Zod 스키마도 동기화 필요.
3. **geocoding 재검증 시점:** `getSavedSearch()` 응답에 `geocodingStatus`를 포함할지, 아니면 클라이언트에서 별도 geocoding 호출 후 상태를 관리할지 — QRY-SAVE-001 작업 시 확정. DTO는 유연하게 optional로 설계.
