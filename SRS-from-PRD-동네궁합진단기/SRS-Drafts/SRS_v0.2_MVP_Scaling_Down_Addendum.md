# SRS Addendum: 1인 개발 기반 MVP 최적화 및 제약사항 조정 명세서

**문서 목적:** 본 문서는 기존 `SRS_v0.1_opus.md`가 요구하는 엔터프라이즈급 개발 복잡도와, '초급 개발 지식을 가진 1인 기획자'의 역량 및 '완전 무료 인프라 기반 운영'이라는 현실적인 조건 사이의 기술적 모순을 해소하기 위한 **공식 추가 명세서(Addendum)**입니다. 본 문서의 내용은 기존 기능 요구사항(FR) 및 비기능 요구사항(NFR)에 우선하여 적용(Override)됩니다.

---

## 1. 구현 가능한 개발 수준으로의 기능 요구사항 덜어내기 (Scope-Down)

AI(Vibe-coding)와 제한된 엔지니어링 지식을 바탕으로 기한 내 제품을 론칭하기 위해, 극도의 복잡성을 유발하는 기능들은 단순화하거나 제거합니다.

### 1.1 외부 API 폴백 구조 해체 (CON-07 / EXT-01~10 조정)
* **기존:** 교통 API 4중 폴백(카카오 → 네이버 → ODsay → 로컬 캐시 DB)
* **조정안:** **카카오 모빌리티 한 개 단일화** 
  * 로컬 DB 사전 캐시 로직과 폴백 전환을 완전히 삭제합니다. API 서버 오류 발생 시 클라이언트 단에서 "일시적 네트워크 지연" 토스트를 띄우어 사용자 행동을 제어(단순 Retry)하는 것으로 마감합니다.

### 1.2 "데드라인 모드"의 매물 크롤링 제거 (REQ-FUNC-016 대체)
* **기존:** 4시간 단위 서버사이드 크롤링(Cron 잡) → 교집합 필터링 → 자체 서비스 내 매물 리스트업 표시
* **조정안:** **매물 플랫폼 아웃링크 연동 체계로 변경**
  * 직접 크롤링하는 기능을 통째로 제거합니다. 
  * "네이버 부동산 바로가기"처럼 동네 조건과 면적, 예산 등 필터 조건을 쿼리 파라미터 셋으로 조합하여 외부 서비스의 검색 결과 URL(새 창 열기)로 랜딩 시킵니다.

### 1.3 PDF 파일 자동 생성 모듈 제거 (REQ-FUNC-023 대체)
* **기존:** 서버사이드 렌더링을 통한 완전한 PDF 생성 및 다운로드 (3초 이내 응답)
* **조정안:** **브라우저 기본 네이티브 인쇄(window.print) + CSS Media 지원**
  * `@react-pdf/renderer`와 같은 별도 서버 사이드 라이브러리를 쓰지 않습니다. 
  * `CSS @media print` 쿼리를 적용하여 화면 상의 불필요한 내비게이션 바 등만 가린 채 사용자에게 브라우저 자체 PDF 저장 창을 띄웁니다.

### 1.4 공공데이터 실시간 호출 중단 (REQ-FUNC-022 조정)
* **기존:** 경찰청 치안 API 등을 실시간으로 호출하고 오버레이
* **조정안:** **정적 파일 에셋(Static JSON/CSV) 임베딩**
  * API 타임아웃 방지를 위해 공공데이터는 기획자가 개발 전 오픈소스 툴(파이썬 등)을 통해 JSON으로 한 번만 추출한 후 프론트엔드 레포지토리에 반영(Commit)합니다.

## 2. 완전 무료 인프라와 비기능 요구사항 간의 기술적 모순 해소

Vercel Hobby(무료), Supabase Free 버전에서 목표 SLA 99.5%, O(N)의 대량 트래픽 교차 연산은 **Vercel Serverless Function Timeout(10초)** 및 병목으로 이어집니다. 이를 해소할 기술 방향을 강제합니다.

### 2.1 서버 함수 타임아웃 모순 해소 : Client-Heavy 연산 구조 전환
기존에는 Vercel의 백엔드(Server Action)에서 카카오 API를 여러 동네에 대해 수십 번 루프를 도는 형태를 상정했으나, 무료 제한(10초)을 회피할 수 없습니다.
* **해결책:** 연산 주도권을 '클라이언트 브라우저'로 이관합니다. 
* **설계 방침:**
  1. Next.js 서버리스 함수는 단일 경로(1:1) 탐색 데이터만 간단히 프록시로 넘기거나 위경도 매핑만 담당.
  2. 프론트엔드(클라이언트 컴포넌트)에서 브라우저 자원을 이용해 Promise.all()로 비동기 동시 다발 호출을 돌립니다. 브라우저는 타임아웃 제한이 넉넉합니다.
  3. 교차점 점수 계산 및 필터링 등 CPU 집약 로직은 서버가 아닌 사용자의 모바일 브라우저 내부(Javascript)에서 수행하도록 로직을 위임합니다.

### 2.2 비기능 요구사항(NFR) 제한 완화
무료 티어에서는 과도한 트래픽 순간 런타임 제어가 불가하므로, SLA 달성 등 엔터프라이즈급 책임 지표들을 MVP 현실에 맞게 내립니다.

| Requirement ID | 원본(기존) 요구 수치 제한 | 조정된 한계 수치(MVP 타협안) | 사유 / 해결 방향 |
| --- | --- | --- | --- |
| **REQ-NF-001** | 응답 시간 p95 ≤ 3,000ms | **p95 ≤ 8,000ms 유지** | 서버사이드 타임아웃 회피를 위해 브라우저에서 클라이언트 기반 비동기 계산 시 로딩 시간 증가 불가피 |
| **REQ-NF-011** | 가용성(Uptime) 99.5% 유지 | **Best Effort** | Vercel Free / Supabase Free 티어의 내재적 서비스 일시정지(Cold Start 등) 제약 감내 |
| **REQ-NF-014** | 재해 복구 시간(RTO) 4시간 | **(항목 폐기)** | 무료 DB 운영 시 자동 복구 드릴 보장 불가. (일 단위 전체 DB 수동 덤프 방식으로 대체) |
| **REQ-NF-039** | 10배 확장 트래픽 처리 설계 보장 | **동시접속 50명, 주간 100건 가이드** | Supabase 커넥션 풀링 과부하 방지를 위해 완전 무료 티어 내 동시 처리 최소화 보장 |
| **REQ-NF-040** | 교통 API 24시간 내 어댑터 전환 설계 | **(항목 폐기)** | 단일 API 올인 전략으로 인터페이스 추상화 등 개발 공수 사전 배제 |

### 2.3 스토리지 및 트래픽 운영 방침 (무료 유지 전략)
* **캐시 데이터 적재 전면 금지:** 기존 API 폴백 용으로 상정된 24h 사전 캐시 DB, 공공 API 사전 수집 크롤링 배치 DB 구조를 모두 삭제합니다. 이로써 Supabase Free(500MB 한도)를 오직 **유저 인증(Session), 공유 결제 로그, 사용 이력(History) 핵심 메타데이터** 용도로만 깔끔히 사용하고 비용 발생 리스크를 완전히 제거합니다.

## 3. 요약: 1인 AI 혼합 개발 워크플로 가이드라인

위 조정안에 따라 향후 Vibe-coding 작업을 프롬프팅할 때 지켜야 할 **골든 룰(Golden Rules)**입니다.

1. *"이 기능은 Vercel Server Function에서 돌리지 말고 무조건 'use client'가 붙은 브라우저 쪽에서 연산하게 해줘"* 라고 명령하세요.
2. 외부 서비스(결제, 지도, 로그)를 붙일 단일 모듈 외에는 앱 구조를 단순 CRUD로 쪼개달라고 지시하세요.
3. *'가장 완벽한 폴백 방어, 에러 핸들링'* 보다 단순 실패 시 *'Alert 창 하나 주고 초기화'*를 MVP 개발 마인드셋의 기본값으로 두십시오.
