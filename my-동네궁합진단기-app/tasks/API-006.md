## 1. 🎯 Summary

- **기능명:** [API-006] 공통 에러 코드 체계 정의 (HTTP Status + Application Error Code + 사용자 메시지 매핑)
- **목적 (Why):**
  - **비즈니스:** 모든 API 엔드포인트(API-001~007)가 일관된 에러 응답 구조를 사용하도록 통합 에러 코드 체계를 확립한다. 5개 도메인(Auth/Diagnosis/ShareLink/SavedSearch/Common)의 에러 코드를 3-Layer(HTTP/appCode/userMessage)로 매핑하여 클라이언트·서버·모니터링 간 에러 처리를 단일화한다.
  - **사용자 가치:** 모든 에러 상황에서 한국어 사용자 메시지가 일관되게 표시되고, Sentry에 구조화된 에러 정보가 전달되어 문제 해결 속도를 높인다.
- **범위 (What):**
  - ✅ 만드는 것: `AppErrorCode` 유니온 타입, `AppErrorMapEntry` 인터페이스, `APP_ERROR_MAP` 통합 매핑, `createAppError()` 헬퍼, `CommonErrorCode` 신규 정의, 배치 1·2의 4개 도메인 ERROR_MAP 통합
  - ❌ 만들지 않는 것: **PaymentErrorCode (절대 금지)**, NextAuth.js 관련 에러, AES-256 관련 에러
- **복잡도:** M
- **Wave:** 1 (API Contract 트랙 — 외부 의존 없음)

- [ ] **3.4** `lib/constants/app-error-map.ts`에 5개 도메인 통합 `APP_ERROR_MAP` 정의
  ```typescript
  import type { AppErrorCode, AppErrorMapEntry } from '@/lib/types/errors';
  import { AUTH_ERROR_MAP } from './auth-errors';
  import { DIAGNOSIS_ERROR_MAP } from './diagnosis-errors';
  import { SHARE_LINK_ERROR_MAP } from './share-link-errors';
  import { SAVED_SEARCH_ERROR_MAP } from './saved-search-errors';
  import { COMMON_ERROR_MAP } from './common-errors';

  // 기존 도메인 MAP → AppErrorMapEntry 변환 어댑터
  function adaptLegacyMap<T extends string>(
    map: Record<T, { message: string; httpStatus: number }>,
    defaultSentryLevel: 'error' | 'warning' | 'info' = 'error'
  ): Record<T, AppErrorMapEntry> {
    const adapted = {} as Record<T, AppErrorMapEntry>;
    for (const [code, entry] of Object.entries(map) as [T, { message: string; httpStatus: number }][]) {
      adapted[code] = {
        httpStatus: entry.httpStatus,
        appCode: code,
        userMessage: entry.message,
        sentryLevel: entry.httpStatus >= 500 ? 'error' : entry.httpStatus >= 400 ? 'warning' : defaultSentryLevel,
      };
    }
    return adapted;
  }

  export const APP_ERROR_MAP: Record<AppErrorCode, AppErrorMapEntry> = {
    ...adaptLegacyMap(AUTH_ERROR_MAP),
    ...adaptLegacyMap(DIAGNOSIS_ERROR_MAP),
    ...adaptLegacyMap(SHARE_LINK_ERROR_MAP),
    ...adaptLegacyMap(SAVED_SEARCH_ERROR_MAP),
    ...COMMON_ERROR_MAP,
  } as Record<AppErrorCode, AppErrorMapEntry>;
  ```

- [ ] **3.5** `lib/helpers/app-error.ts`에 `createAppError()` 헬퍼 작성
  ```typescript
  import type { AppErrorCode, AppErrorDTO, AppErrorMapEntry } from '@/lib/types/errors';
  import { APP_ERROR_MAP } from '@/lib/constants/app-error-map';

  export function createAppError(code: AppErrorCode, detail?: string): AppErrorDTO {
    const entry: AppErrorMapEntry = APP_ERROR_MAP[code];
    return {
      code,
      message: entry.userMessage,
      httpStatus: entry.httpStatus,
      detail,
      sentryLevel: entry.sentryLevel,
    };
  }

  /** Next.js Route Handler에서 사용하는 JSON Response 변환 */
  export function toErrorResponse(code: AppErrorCode, detail?: string): Response {
    const error = createAppError(code, detail);
    return Response.json(
      { error: { code: error.code, message: error.message, detail: error.detail } },
      { status: error.httpStatus }
    );
  }
  ```

- [ ] **3.6** `lib/helpers/sentry-error.ts`에 Sentry 연동 에러 리포팅 유틸리티 작성
  ```typescript
  import * as Sentry from '@sentry/nextjs';
  import type { AppErrorDTO } from '@/lib/types/errors';

  export function reportErrorToSentry(error: AppErrorDTO, originalError?: Error): void {
    const level = error.sentryLevel ?? 'error';
    Sentry.withScope((scope) => {
      scope.setLevel(level);
      scope.setTag('error_code', error.code);
      scope.setExtra('httpStatus', error.httpStatus);
      scope.setExtra('userMessage', error.message);
      if (error.detail) scope.setExtra('detail', error.detail);
      if (originalError) {
        Sentry.captureException(originalError);
      } else {
        Sentry.captureMessage(`[${error.code}] ${error.message}`);
      }
    });
  }
  ```

## 4. ✅ Acceptance Criteria (GWT 패턴, BDD)

**AC-1 (정상):** createAppError()로 에러 DTO 생성 성공
- **Given** `CommonErrorCode.INVALID_INPUT`과 `detail: '주소 형식 오류'`
- **When** `createAppError(CommonErrorCode.INVALID_INPUT, '주소 형식 오류')` 호출
- **Then** 반환된 `AppErrorDTO`의 `code`가 `'COMMON_INVALID_INPUT'`, `httpStatus`가 `400`, `message`가 한국어, `detail`이 `'주소 형식 오류'`

**AC-2 (예외):** 존재하지 않는 에러 코드로 MAP 조회 시 TypeScript 에러
- **Given** `AppErrorCode` 유니온에 포함되지 않는 문자열 `'INVALID_CODE'`
- **When** `createAppError('INVALID_CODE' as any)` 호출
- **Then** TypeScript 컴파일 타임에 타입 에러 발생 (런타임에서는 undefined 반환)

**AC-3 (예외):** Payment 도메인 에러 코드 절대 미포함
- **Given** `APP_ERROR_MAP`의 모든 키
- **When** `PAYMENT` 또는 `PAY` 포함 키 필터링
- **Then** 결과 0건

**AC-4 (경계):** 배치 1·2 도메인 에러 코드 전수 매핑 검증
- **Given** Auth(9개) + Diagnosis(8개) + ShareLink(7개) + SavedSearch(5개) + Common(5개) = 34개 에러 코드
- **When** `Object.keys(APP_ERROR_MAP).length` 확인
- **Then** 34개 이상 (도메인별 확장 가능)

**AC-5 (정합성):** 배치 1·2의 4개 ERROR_MAP과 통합 구조 일치
- **Given** API-001의 `AUTH_ERROR_MAP`, API-002의 `DiagnosisErrorCode`, API-003의 `SHARE_LINK_ERROR_MAP`, API-005의 `SAVED_SEARCH_ERROR_MAP`
- **When** 각 MAP의 에러 코드가 `APP_ERROR_MAP`에 존재하는지 확인
- **Then** 모든 기존 에러 코드가 `APP_ERROR_MAP`에 포함되며, `userMessage`가 기존 `message`와 동일

| API-002.md | 동일 — `DIAGNOSIS_ERROR_MAP`에 `sentryLevel` 미포함. 어댑터로 처리 가능 | L |
| API-003.md | 동일 — `SHARE_LINK_ERROR_MAP`에 `sentryLevel` 미포함. 어댑터로 처리 가능 | L |
| API-005.md | 동일 — `SAVED_SEARCH_ERROR_MAP`에 `sentryLevel` 미포함. 어댑터로 처리 가능 | L |

> 위 4개 ISSUE 모두 `adaptLegacyMap()` 어댑터로 하위 호환 처리되므로 코드 변경 없이 통합 가능. 향후 각 도메인 MAP을 `AppErrorMapEntry` 구조로 직접 마이그레이션하면 어댑터 제거 가능.

### 기타 보류 사항

1. **에러 코드 i18n 확장:** 현재 `userMessage`는 한국어만 지원. 다국어 지원 시 `userMessage`를 i18n 키로 교체하는 구조 변경 필요 — v2 이후 검토.
2. **에러 코드 numbering 체계:** 현재 문자열 기반 (`AUTH_OAUTH_FAILED` 등). 숫자 기반 에러 코드가 필요한지 — MVP에서는 문자열 유지.
