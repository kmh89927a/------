## 1. 🎯 Summary

- **기능명:** [API-007] 카카오 모빌리티 API 클라이언트 인터페이스 정의 (KakaoTransportClient 타입)
- **목적 (Why):**
  - **비즈니스:** 두 동선 교차 진단의 핵심 외부 의존성인 카카오 모빌리티 API와의 통신 계약을 인터페이스로 확립하여, 실제 API 클라이언트와 Mock/Test 구현체를 교체 가능하게 한다.
  - **사용자 가치:** DI(Dependency Injection) 가능한 인터페이스 설계로, 개발·테스트·장애 시 유연하게 구현체를 전환할 수 있어 서비스 안정성이 향상된다.
- **범위 (What):**
  - ✅ 만드는 것: KakaoTransportClient 인터페이스, 요청/응답 타입 (KakaoRouteRequest, KakaoRouteResponse, KakaoTransportResponse), CommuteInfoDTO 변환 인터페이스, 클라이언트 설정 타입 (타임아웃·재시도), 구현체 스텁 (후속 태스크에서 완성)
  - ❌ 만들지 않는 것: 실제 HTTP 클라이언트 구현(후속 CMD-DIAG 태스크), 네이버 지도 API 인터페이스(MVP 제외), 결제 관련 코드

- [ ] **3.4** `lib/external/kakao-transport/mapper.ts`에 카카오 응답 → CommuteInfoDTO 변환 함수 시그니처 + 구현
  ```typescript
  import type { KakaoTransportResponse, KakaoRoute } from './types';
  import type { CommuteInfoDTO } from '@/lib/types/diagnosis';

  /**
   * 카카오 모빌리티 응답을 CommuteInfoDTO로 변환
   * @param kakaoResponse 카카오 API 원본 응답
   * @returns CommuteInfoDTO (API-002 정의)
   */
  export function mapKakaoResponseToCommuteInfo(
    kakaoResponse: KakaoTransportResponse
  ): CommuteInfoDTO {
    const route = kakaoResponse.routes[0];
    if (!route || route.resultCode !== 0) {
      throw new Error(`Kakao route error: ${route?.resultMsg ?? 'No route'}`);
    }

    const hasCarSection = route.sections.some(s => s.type === 'car');
    const transportType: 'transit' | 'car' = hasCarSection ? 'car' : 'transit';

    return {
      durationMinutes: Math.round(route.summary.totalDurationSeconds / 60),
      transportType,
      transfers: route.summary.totalTransfers,
      walkingMinutes: Math.round(route.summary.totalWalkingSeconds / 60),
    };
  }
  ```

## 4. ✅ Acceptance Criteria (GWT 패턴, BDD)

**AC-1 (정상):** 카카오 응답 → CommuteInfoDTO 변환 성공
- **Given** 대중교통 경로 응답 (`totalDurationSeconds: 2100`, `totalTransfers: 1`, `totalWalkingSeconds: 480`)
- **When** `mapKakaoResponseToCommuteInfo(response)` 호출
- **Then** `durationMinutes === 35`, `transportType === 'transit'`, `transfers === 1`, `walkingMinutes === 8`

**AC-2 (예외):** 경로 없음 응답 시 에러
- **Given** `routes[0].resultCode !== 0` (경로 없음)
- **When** `mapKakaoResponseToCommuteInfo(response)` 호출
- **Then** Error throw, 메시지에 `resultMsg` 포함

**AC-3 (예외):** 빈 routes 배열 시 에러
- **Given** `routes: []` (빈 배열)
- **When** `mapKakaoResponseToCommuteInfo(response)` 호출
- **Then** Error throw

**AC-4 (경계):** 자차 경로 변환
- **Given** sections에 `type: 'car'`만 포함된 응답
- **When** `mapKakaoResponseToCommuteInfo(response)` 호출
- **Then** `transportType === 'car'`, `transfers === 0`, `walkingMinutes === 0`

**AC-5 (경계):** 타임아웃·재시도 설정값 검증
- **Given** `DEFAULT_KAKAO_CONFIG` 상수
- **When** `timeoutMs`와 `maxRetries` 확인
- **Then** `timeoutMs === 5000` (REQ-FUNC-007), `maxRetries === 1` (REQ-FUNC-007)

2. **카카오 모빌리티 API 실제 응답 구조:** 현재 타입은 SRS §3.1 EXT-01과 §6.7 CLD 기반으로 정의. 실제 카카오 API 문서(https://developers.kakaomobility.com)와 응답 구조가 다를 수 있음 — CMD-DIAG-001에서 실제 API 호출 시 정렬.
3. **대중교통 vs 자차 API 엔드포인트 분리:** 카카오 모빌리티 API는 대중교통(`/v1/directions/transit`)과 자차(`/v1/directions`) 엔드포인트가 분리됨. `KakaoRouteRequest.transportType` 필드로 분기 — 실제 구현(CMD-DIAG-001)에서 확정.
4. **Client Component에서의 API Key 노출:** REQ-FUNC-003에 따라 교통 API 호출이 Client Component에서 이루어짐. 브라우저에서 API Key가 노출되는 보안 이슈 — Supabase Edge Function 프록시 또는 Next.js API Route를 프록시로 사용하는 방안 검토 필요.
5. **네이버 지도 API 인터페이스:** 현재 MVP에서는 카카오만 사용하지만, v1.5에서 네이버 지도 API 폴백이 필요할 수 있음. `ITransportClient` 상위 인터페이스로 추상화하는 방안은 후속 태스크에서 검토.
