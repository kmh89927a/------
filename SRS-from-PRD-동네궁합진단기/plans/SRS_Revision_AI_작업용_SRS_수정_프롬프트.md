# AI 작업용 SRS 수정 프롬프트 (Prompt Plan)

이 문서는 기존 `SRS_v0.1_opus.md` 파일의 과도한 스펙을 MVP 현실(1인 기획자/완전 무료 인프라)에 맞게 **AI에게 원본 문서 수정을 직접 명령할 때 복사-붙여넣기 할 수 있는 '수정용 지시어(프롬프트)' 모음**입니다. 

Cursor, GitHub Copilot, 또는 Claude 등의 AI 프롬프트 창에 아래의 Step 순서대로 프롬프트를 복사하여 붙여넣고 실행을 지시하세요.

---

### Step 1: 아키텍처 제약 및 외부 시스템 폴백(Fallback) 해체
**목표:** 4중 폴백 및 사전 캐시 DB, 크롤링 DB 등 복잡한 서버 데이터를 모두 날리고 가장 단순한 상태로 만듭니다.

**📋 AI에 복사해 넣을 프롬프트:**
```text
@SRS_v0.1_opus.md 파일을 분석하고 다음 규칙에 따라 '1.2.3 Constraints (제약사항)' 및 '3.1 External Systems'를 수정해주세요.

1. 제약사항 CON-07(교통 데이터 4중 백업 체계 및 사전 캐시 DB)을 완전히 제거하고, "CON-07: 교통 API는 카카오 모빌리티 1종만 사용하며, 장애 시 클라이언트 화면에 단순 에러 모달만 노출한다"로 변경하세요.
2. Section '3.1 External Systems'에서 EXT-02 (네이버 지도 백업), EXT-09 (ODsay), EXT-10 (TAGO) 행을 완전히 삭제하세요. 
3. '3.1.1 외부 시스템 장애 시 우회 전략' 표에서 "폴백" 및 "사전 적재 DB", "캐시 DB" 내용을 모두 삭제하세요.
4. EXT-08(매물 데이터 크롤링) 항목을 삭제하고, 매물 연동은 "네이버 부동산 아웃링크 (조건 파라미터 결합 이동)" 방식으로 변경한다고 명세하세요.
```

---

### Step 2: 클라이언트 중심 연산 변환 및 데드라인/리포트 기능 간소화
**목표:** Vercel 서버 10초 타임아웃 방지를 위해 무거운 백엔드 Server Action 로직을 모바일 브라우저 Client Side로 넘기고 PDF, 크롤링 잡을 걷어냅니다.

**📋 AI에 복사해 넣을 프롬프트:**
```text
@SRS_v0.1_opus.md 파일의 '4.1 Functional Requirements' 섹션을 다음 요구에 맞게 전면 수정하세요.

1. "데드라인 모드 (F3)" 섹션:
   - REQ-FUNC-016 (급매 데이터 크롤링/Cron Job) 항목을 삭제하고, "교집합 동네를 클릭하면 해당 조건을 네이버 부동산 검색 URL 파라미터로 조합하여 아웃링크로 새 창을 연다"는 단순 랜딩 요구사항으로 대체하세요.
   - REQ-FUNC-017에서 서버 단위 교집합 매물 필터링 내용을 지우세요.
2. "싱글 모드 간소화 리포트 (F4)" 섹션:
   - REQ-FUNC-023 의 서버사이드를 활용한 PDF 다운로드 기능을 지우고, "클라이언트 브라우저의 기본 window.print() 메서드 호출과 CSS @media print 제어를 통해 PDF 저장을 안내한다"로 변경해 주세요.
3. "두 동선 교차 진단 (F1)" 섹션: REQ-FUNC-003에 "Vercel 무료 티어의 10초 Timeout을 방지하기 위해, 외부 교통 API 반복 호출 연산과 교차 연산 로직은 Next.js 서버(Server Action)가 아닌, 사용자 브라우저(Client Component) 내부에서 비동기 병렬 구조(Promise.all)로 처리해야 한다"는 설계 제약을 명시하세요.
```

### Step 3: 무료 인프라 모순 해결을 위한 비기능 요구사항(NFR) 및 KPI 다이어트
**목표:** 엔터프라이즈급 달성 목표(SLA, 확장성)를 삭제하고 완전 무료 티어 제약 안으로 숨통을 틉니다.

**📋 AI에 복사해 넣을 프롬프트:**
```text
@SRS_v0.1_opus.md 파일의 '4.2 Non-Functional Requirements'를 다음과 같이 대폭 삭제/조정하세요.

1. '4.2.1 성능': REQ-NF-001의 응답 시간 목표를 "p95 ≤ 3,000ms"에서 "p95 ≤ 8,000ms (클라이언트 API 콜 기준)"으로 수정하세요.
2. '4.2.2 가용성 및 신뢰성': 
   - REQ-NF-011(SLA 99.5%)를 "Best Effort 지원 (무료 티어 제약 수용)"으로 변경하세요.
   - REQ-NF-014(RTO 4시간), REQ-NF-015(RPO 1시간)를 MVP 인프라(무료 Supabase)에 맞지 않으므로 과감하게 삭제(Strike-through 처리 또는 릴리스 대상 제외)하세요.
3. '4.2.7 Scalability': REQ-NF-039(10배 수평 확장 대응)와 REQ-NF-040(교통 API 24시간 내 어댑터 전환) 요구사항을 삭제하세요 (1차 MVP 스코프 아님).
```

### Step 4: API Overview 및 ERD 다이어트 방영
**목표:** 없어진 서버 사이드 크롤링 잡 및 복잡 캐시 테이블을 데이터 모델 구성도(ERD)에서 치워버립니다.

**📋 AI에 복사해 넣을 프롬프트:**
```text
@SRS_v0.1_opus.md 파일의 '6. Appendix' 섹션을 다음과 같이 최종 업데이트하세요.

1. '6.1 API Endpoint List'에서 `API-11` (PDF 다운로드 Route Handler) 및 `API-12` (매물 크롤링 Cron Job)를 완전히 삭제하세요. 
2. '6.2 Entity & Data Model (ERD)': "CANDIDATE_AREA" 등의 캐시 성격 테이블을 제거하고 오직 USER, DIAGNOSIS, SHARE_LINK, PAYMENT 기록 등 상태를 관리하기 위해 꼭 필요한 핵심 요소만 남기도록 ERD 항목 설명을 축소 조정해 주세요.
```
