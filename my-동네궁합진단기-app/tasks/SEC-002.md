## 1. 🎯 Summary

- **기능명:** [SEC-002] Next.js Middleware Rate Limiting — IP당 분당 60req 차단 (Upstash Redis 기반)
- **목적 (Why):**
  - **비즈니스:** 배치 6의 CMD-SHARE-004에서 in-memory Map으로 구현된 Rate Limit의 서버리스 호환 한계를 해소한다. Upstash Redis 기반 글로벌 Rate Limit 인프라를 Middleware 레벨에서 제공하여, 모든 API 및 공유 링크 경로를 보호한다.
  - **사용자 가치:** 악성 트래픽(DDoS, 무차별 대입)으로부터 서비스를 보호하여 정상 사용자의 서비스 가용성을 유지한다.
- **범위 (What):**
  - ✅ 만드는 것: `middleware.ts` Rate Limit 로직, `lib/security/rate-limit.ts` (Upstash Ratelimit), Upstash Redis 프로비저닝 가이드, X-RateLimit-* 응답 헤더, 429 응답 처리, 정적 자산/헬스체크 제외 matcher
  - ❌ 만들지 않는 것: WAF 구성, 결제 코드, NextAuth.js, AES-256 암호화
- **복잡도:** M
- **Wave:** 1 (보안 트랙)

  ```typescript
  // lib/security/rate-limit.ts
  import { Ratelimit } from '@upstash/ratelimit';
  import { Redis } from '@upstash/redis';

  const redis = Redis.fromEnv();

  // REQ-NF-022: IP당 분당 60req — Sliding Window
  const ratelimit = new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(60, '1 m'),
    analytics: true,
    prefix: 'ratelimit:global',
  });

  export interface RateLimitResult {
    success: boolean;
    limit: number;
    remaining: number;
    reset: number; // Unix timestamp (seconds)
  }

  export async function checkRateLimit(identifier: string): Promise<RateLimitResult> {
    const result = await ratelimit.limit(identifier);
    return {
      success: result.success,
      limit: result.limit,
      remaining: result.remaining,
      reset: Math.ceil(result.reset / 1000), // ms → seconds
    };
  }
  ```

- [ ] **3.4** `middleware.ts`에 Rate Limit 로직 통합
  ```typescript
  // middleware.ts
  import { NextResponse, type NextRequest } from 'next/server';
  import { checkRateLimit } from '@/lib/security/rate-limit';

  export async function middleware(req: NextRequest) {
    // IP 추출 (Vercel의 x-forwarded-for 헤더 활용)
    const ip = req.ip ?? req.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ?? 'unknown';

    // Rate Limit 체크
    const { success, limit, remaining, reset } = await checkRateLimit(ip);

    if (!success) {
      return new NextResponse(
        JSON.stringify({
          error: {
            code: 'COMMON_RATE_LIMITED',
            message: '요청이 너무 많습니다. 잠시 후 다시 시도해 주세요.',
          },
        }),
        {
          status: 429,
          headers: {
            'Content-Type': 'application/json',
            'X-RateLimit-Limit': String(limit),
            'X-RateLimit-Remaining': '0',
            'X-RateLimit-Reset': String(reset),
            'Retry-After': String(Math.max(0, reset - Math.ceil(Date.now() / 1000))),
          },
        }
      );
    }

    // 정상 요청 → RateLimit 헤더 추가
    const response = NextResponse.next();
    response.headers.set('X-RateLimit-Limit', String(limit));
    response.headers.set('X-RateLimit-Remaining', String(remaining));
    response.headers.set('X-RateLimit-Reset', String(reset));

    return response;
  }

  // 보호 대상 경로 — 정적 자산·헬스체크 제외
  export const config = {
    matcher: [
      '/api/:path*',
      '/share/:path*',
      '/((?!_next/static|_next/image|favicon.ico|api/health|.*\\.(?:svg|png|jpg|jpeg|gif|webp|ico)$).*)',
    ],
  };
  ```

## 4. ✅ Acceptance Criteria (GWT 패턴, BDD)

**AC-1 (정상):** 60req 이내 → 정상 통과
- **Given** 동일 IP에서 1분 내 59회 요청 완료
- **When** 60번째 `/api/diagnosis/123` 요청
- **Then** HTTP 200 응답, `X-RateLimit-Remaining: 0` 헤더 포함

**AC-2 (경계 ⭐):** 61번째 요청 → 429 응답
- **Given** 동일 IP에서 1분 내 60회 요청 완료
- **When** 61번째 요청
- **Then** HTTP 429 응답, body에 `COMMON_RATE_LIMITED` 에러 코드, `X-RateLimit-Remaining: 0`, `Retry-After` 헤더 포함

**AC-3 (예외):** 정적 자산은 Rate Limit 미적용
- **Given** `/_next/static/...`, `/favicon.ico`, `*.png` 등 정적 자산 경로
- **When** 해당 경로로 100회 이상 요청
- **Then** 모든 요청 정상 통과, `X-RateLimit-*` 헤더 미포함

**AC-4 (보안):** IP 추출 — x-forwarded-for 헤더 활용
- **Given** Vercel 배포 환경에서 `req.ip`가 사용 가능
- **When** 클라이언트 요청
- **Then** `req.ip` 또는 `x-forwarded-for` 첫 번째 값으로 IP 식별, IP별 독립 Rate Limit 적용

**AC-5 (모니터링):** Rate Limit 초과 시 Sentry breadcrumb 기록
- **Given** Rate Limit 초과 이벤트 발생
- **When** Sentry breadcrumb 데이터 확인
- **Then** `category: 'rate-limit'`, `level: 'warning'`, IP 일부 마스킹

**AC-6 (정합성):** CMD-AUTH-003 Middleware와 공존
- **Given** Rate Limit + Auth 세션 검증이 동일 `middleware.ts`에 존재
- **When** 인증된 사용자가 요청
- **Then** Rate Limit 체크 후 Auth 세션 검증 순서로 실행, 양쪽 모두 통과 시 NextResponse.next()

## 9. 🚧 Open Questions / Risks (보류 사항)

1. **CMD-SHARE-004 마이그레이션 follow-up ⭐:** CMD-SHARE-004의 in-memory Rate Limit(`lib/share/rate-limiter.ts`)은 본 ISSUE 완성 후 `@/lib/security/rate-limit`의 Upstash 인프라로 마이그레이션 필요. 별도 미니 태스크로 트래킹.

2. **Upstash vs Vercel KV 트레이드오프:**
   | 항목 | Upstash Redis | Vercel KV |
   |---|---|---|
   | 무료 티어 | 10,000 cmd/day | 30,000 requests/month |
   | REST API | ✅ (Edge 호환) | ✅ (Edge 호환) |
   | 글로벌 분산 | ✅ | ✅ |
   | 비용 | 종량제 합리적 | Vercel Pro 필요 시 비용 증가 |
   → **본 ISSUE는 Upstash Redis 권장** (무료 티어 풍부 + REST API + 글로벌 분산)

3. **인증된 사용자 우대 정책:** 인증된 사용자에게 더 높은 Rate Limit (예: 120req/분)을 적용할지 — MVP에서는 일률 60req/분 적용. Open Beta 이후 검토.

4. **Upstash Free Tier 한계:** 10,000 cmd/day 제한. MVP 트래픽(50건/주 진단)에서 충분하나, 확장 시 유료 전환 필요.

5. **Edge Runtime 호환:** Upstash Redis는 REST API 기반이므로 Next.js Edge Middleware에서 사용 가능. Node.js runtime 제약 없음.
