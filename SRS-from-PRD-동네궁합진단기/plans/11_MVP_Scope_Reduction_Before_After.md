# 1인 개발 MVP 스코프 축소 비교 요약 (Before / After)

본 문서는 기존의 방대한 v1 기획/명세 문서들을 **1인 초급 개발자(Next.js+Supabase 스택)**가 단기간 내 구현할 수 있는 **MVP(v1.1 / v1.5) 스코프**로 축소하면서 발생한 주요 변경 사항을 파일별로 비교한 내용입니다.

---

## 1. 요구사항 명세서 (SRS)
- **변경 파일:** `05_SRS_v1.md` ➡️ `05_SRS_v1.5.md`

| 구분 | Before (v1.0 전체 스코프) | After (v1.5 MVP 스코프) | 축소 사유 / 기대 효과 |
| :--- | :--- | :--- | :--- |
| **F5. 입력값 저장** | **복합 재탐색:** 99.9% 성공 보장. 비교 뷰(항목 5개 이상), 시나리오 최대 3개 동시 비교. 행정동 변경 자동 감지(95% 정확도). | **간이 저장·불러오기:** 세션 종료 시 best effort 저장. geocoding 실패 시 "다시 입력" 안내. 비교·시나리오 기능 삭제. | 복잡한 UI/UX 구현 공수 제거. 백엔드 재계산 로직 제거. |
| **보안 (Security)** | **AES-256 PII 암호화** 저장 필수. OWASP Top 10 DAST 스캔 (ZAP). | Supabase 기본 전송 암호화(**TLS/SSL**) 활용. AES 및 DAST 요구사항 제거. | 보안 감사 인프라 설정 비용 및 시간 대폭 단축. |
| **모니터링 (NFR)** | 커스텀 슬랙 알림 임계치, Datadog/Amplitude 복합 알림. 퍼널 이상 감지 통보. | **Sentry 기본 연결** 수준으로 완화. 수동 및 기본 알림 채택. | 오버엔지니어링 방지 및 1인 운영 오버헤드 최소화. |
| **데이터 모델 (ERD)** | User, Diagnosis 연계 등 다수 테이블 + `DongMap`(매핑), `ViewLog`(로그) 엔터티 존재 | `SavedSearch` 3개 필드 단순화. `ViewLog`, `DongMap` 제거. | 스키마 마이그레이션 단순화 및 서버 쿼리 최적화. |
| **코드베이스 정리** | 해당 기능 관련 UseCase, 시퀀스 다이어그램, API 명세 다수 존재. | 관련 API(`replaySearch`), 테스트 시나리오(TC-026~028), NFR 항목 일괄 삭제 (취소선 후 삭제). | 문서의 정합성 유지 및 SSOT(단일 진실 공급원) 확보. |

## 2. 제품 요구사항 정의서 (PRD)
- **변경 파일:** `00_PRD_v1.md` ➡️ `00_PRD_v1.1.md`

| 구분 | Before (rev.2) | After (rev.3) | 축소 사유 / 기대 효과 |
| :--- | :--- | :--- | :--- |
| **MoSCoW 우선순위** | `S` 이전 입력값 자동 저장 + 원클릭 재탐색 + 발령 알림 | `M` 이전 입력값 자동 저장 + 불러오기 (간이) | 백엔드 재계산 배제 / 폼만 다시 채우는 방식으로 MVP 합류. |
| **Story 3-5 AC 표** | AC-1(99.9% 보장), AC-2(비교뷰), AC-3(시나리오), AC-N1(자동 감지) | AC-1(best effort, 실패 시 안내) 하나만 유지 | 과업 난이도 대폭 하락. |
| **보안 기준 (§5-3)** | AES-256 암호화, ISMS-P 가이드 준수, OWASP DAST 기준 통과 | Supabase 전송 보안 의존, 취약점 스캔 항목 배제 | 초기 런칭 시점의 마찰(Friction) 최소화. |
| **Out of Scope** | 매물 중개, 셔틀 DB, 비수도권 등 3건 | 과거 vs 현재 비교 뷰, 시나리오 복수 비교, 행정동 감지 추가 **(총 6건)** | 향후 개발(v1.5/v2.0)로 연기 명문화. |
| **리스크 완화(R4)** | 열람 비밀번호 세팅 + 열람 기록 알림 발송 | 링크 만료 30일 + UUID v4 entropy 의존. 비밀번호/알림은 추후 검토. | 개발 티켓(CMD-SHARE-004 등) 리소스 절감. |

### 💡 요약 결론
- **스코프 다이어트:** 오버엔지니어링(엔터프라이즈급 보안, 복잡한 비교 연산, 잦은 Cron 업데이트)을 걷어냈습니다.
- **방향성 재조정:** 1인 개발자가 가장 빠르고 합리적으로 **지표(Product-Market Fit)**를 테스트할 수 있게끔 가벼운 버전을 채택하였습니다.
- **문서 간 정합성 보장:** PRD(요구사항) ↔ SRS(명세) ↔ Task(이슈) ↔ Plan(일정) 에 걸쳐 일관성 있게 삭제/축소 항목의 싱크(Sync)를 맞췄습니다.
